# nerve-wml v1.5.2 — docs refresh to v1.5.1

**Release date:** 2026-04-21
**Scope:** documentation patch.

## Why a release just for docs

v1.5.1 shipped on PyPI with a `README.md` that still advertised `Status
— v1.2.3 (2026-04-20)` — the scientific baseline was accurate but the
three feature releases that landed the same day (v1.4.0 plasticity,
v1.5.0 Gumbel + spectrogram + dream-bridge scaffold, v1.5.1 metadata
sync) were invisible on the PyPI landing page. A reviewer arriving on
[pypi.org/project/nerve-wml](https://pypi.org/project/nerve-wml/) would
read "v1.2.3" and infer the project is frozen — the opposite of what
the git log shows.

PyPI renders the `README.md` **bundled with the uploaded wheel**, so a
post-release fix to `README.md` on master is not visible until the next
wheel is published. Hence this micro-release: no code change, just ship
the already-committed README + CHANGELOG sync (master commit `096c51f`)
to PyPI.

## Changed

- `README.md` — `Status` header bumped to `v1.5.1` (now `v1.5.2` in this
  wheel), new "Post-v1.2.3 API additions" table listing the four issues
  (#4, #5, #6, #7) with downstream motivation, `pip install nerve-wml`
  in the Install section. No change to the v1.2.3 scientific baseline
  block — that content is load-bearing and stays verbatim.
- `CHANGELOG.md` — backfilled entries for v1.4.0, v1.5.0, v1.5.1 in
  Keep-a-Changelog format.

## Unchanged

- Every byte of runtime code. 102 tests still pass identically.
- Public API surface, including the three v1.5.0 opt-in knobs.
- Zenodo concept DOI `10.5281/zenodo.19656342` keeps resolving to the
  latest record (this release's version DOI is minted automatically by
  the webhook).

## Upgrade note

`pip install --upgrade nerve-wml` jumps straight from v1.5.1 to v1.5.2.
Importers that pinned `nerve-wml==1.5.1` can stay on that version
indefinitely — there is no behavioural difference.
